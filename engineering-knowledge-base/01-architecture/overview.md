# 三省六部制架构总览

> **版本**: 1.0.0
> **用途**: 多Agent项目自动开发框架架构设计

---

## 一、架构理念

### 1.1 设计原则

基于中国古代三省六部制，结合现代软件工程实践：

```
┌─────────────────────────────────────────────────────────────┐
│ 核心设计原则                                                │
├─────────────────────────────────────────────────────────────┤
│ 1. 分权制衡 - 决策、审核、执行三权分立                      │
│ 2. 强制审核 - 门下省审核不是可选，是必经环节                │
│ 3. 专业分工 - 六部各司其职，术业有专攻                      │
│ 4. 可追溯性 - 每步操作都有记录，可回溯                      │
│ 5. 状态约束 - 任务流转受状态机控制，不可跳过                │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 与传统框架对比

| 特性 | CrewAI | AutoGen | 本框架 |
|------|--------|---------|--------|
| 审核机制 | ❌ 无 | ⚠️ 可选 | ✅ 强制（门下省） |
| 状态机 | ⚠️ 简单 | ⚠️ 简单 | ✅ 完整9状态 |
| 权限控制 | ❌ 无 | ❌ 无 | ✅ 权限矩阵 |
| 知识沉淀 | ⚠️ 弱 | ⚠️ 弱 | ✅ 记忆分层 |
| 干预能力 | ❌ 无 | ⚠️ 有限 | ✅ 实时干预 |

## 三、三省职责

### 3.1 中书省（决策调度层）

| 角色 | 职责 | 输入 | 输出 |
|------|------|------|------|
| 需求分析器 | 理解用户需求，提取关键信息 | 自然语言需求 | 需求文档 |
| 架构设计器 | 设计系统架构，制定技术方案 | 需求文档 | 架构设计 |
| 任务调度器 | 拆解任务，分配给各部门 | 架构设计 | 任务列表 |

**核心能力**：
- 自然语言理解
- 架构设计
- 任务分解
- 资源协调

### 3.2 门下省（监察审核层）

| 角色 | 职责 | 审核内容 | 封驳条件 |
|------|------|----------|----------|
| 质量审核员 | 代码/文档质量 | 代码规范、文档完整性 | 质量不达标 |
| 安全审计员 | 安全漏洞 | 敏感信息、权限设计 | 存在安全隐患 |
| 标准把控员 | 规范一致性 | 架构合理、流程合规 | 偏离规范 |

**核心能力**：
- 代码审查
- 安全扫描
- 规范检查
- 风险识别

**封驳机制**：
```
方案提交 → 门下省审核 → {通过: 准奏, 不通过: 封驳}
                              │
                              ▼（封驳）
                         中书省重新规划
                              │
                              ▼
                         再次提交审核
                              │
                              ▼（循环直到通过或人工干预）
```

### 3.3 尚书省（执行层）

| 角色 | 职责 | 执行内容 |
|------|------|----------|
| 调度官 | 接收准奏任务 | 分配给六部 |
| 协调官 | 监控执行进度 | 处理阻塞、协调资源 |
| 汇总官 | 收集执行结果 | 整合成最终产出 |

**核心能力**：
- 任务派发
- 进度跟踪
- 结果整合
- 异常处理

## 五、状态流转

### 5.1 任务状态机

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│  需求   │────→│  规划   │────→│  审核   │────→│  派发   │
│ (draft) │     │(planning)│    │(review) │     │(dispatch)│
└─────────┘     └─────────┘     └─────────┘     └─────────┘
                                      │               │
                                      │ 封驳          │
                                      ▼               │
                               ┌─────────┐           │
                               │  返工   │           │
                               │(reject) │           │
                               └─────────┘           │
                                                     │
                                                     ▼
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│  完成   │←────│  验收   │←────│  审查   │←────│  执行   │
│ (done)  │     │(accept) │     │(check)  │     │(execute)│
└─────────┘     └─────────┘     └─────────┘     └─────────┘
```

### 5.2 状态说明

| 状态 | 触发条件 | 负责省部 | 可转换状态 |
|------|----------|----------|------------|
| draft | 用户提交需求 | 中书省 | planning |
| planning | 中书省完成规划 | 中书省 | review, reject |
| review | 门下省审核中 | 门下省 | dispatch, reject |
| reject | 门下省封驳 | 中书省 | planning |
| dispatch | 尚书省派发任务 | 尚书省 | execute |
| execute | 六部执行中 | 六部 | check |
| check | 门下省验收中 | 门下省 | accept, execute |
| accept | 验收通过 | 尚书省 | done |
| done | 任务完成 | - | - |

## 七、设计决策

### 7.1 为什么需要门下省强制审核？

```
┌─────────────────────────────────────────────────────────────┐
│ 传统框架的问题                                              │
├─────────────────────────────────────────────────────────────┤
│ ❌ Agent自由协作 → 结果不可控                              │
│ ❌ 无质量关卡 → 质量参差不齐                               │
│ ❌ 事后检查 → 返工成本高                                   │
│ ❌ 无法追溯 → 不知道哪里出了问题                            │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 门下省的价值                                                │
├─────────────────────────────────────────────────────────────┤
│ ✅ 事前审核 → 问题在执行前被发现                            │
│ ✅ 强制关卡 → 每个方案都必须经过审核                        │
│ ✅ 封驳机制 → 不合格直接打回，不浪费执行资源                │
│ ✅ 标准化 → 统一质量标准，减少个人差异                      │
└─────────────────────────────────────────────────────────────┘
```

### 7.2 为什么需要状态机？

1. **防止跳过环节** - 不允许从draft直接到execute
2. **可追溯** - 每个状态变更都有记录
3. **异常恢复** - 知道当前在哪个阶段，便于恢复
4. **并发控制** - 同一时间只允许一个状态
