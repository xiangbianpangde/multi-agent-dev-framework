# 双Agent审查机制

> **版本**: 1.0.0
> **用途**: 复杂任务的双Agent协作审查规范

---

## 一、机制概述

### 1.1 为什么需要双Agent审查？

```
┌─────────────────────────────────────────────────────────────┐
│ 单Agent的问题                                                │
├─────────────────────────────────────────────────────────────┤
│ ❌ 基础模型：能力强但成本高、速度慢                         │
│ ❌ 轻量模型：速度快、成本低，但复杂任务易出错               │
│ ❌ 无审查：错误不易被发现                                   │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│ 双Agent的价值                                                │
├─────────────────────────────────────────────────────────────┤
│ ✅ 成本优化：基础模型做，高性能模型审                       │
│ ✅ 质量保证：独立审查，发现更多问题                         │
│ ✅ 知识传递：审查意见可沉淀为改进点                         │
│ ✅ 风险控制：重要输出必须经过审查                           │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 适用场景

| 场景 | 是否适用 | 原因 |
|------|----------|------|
| 复杂技术文档编写 | ✅ 适用 | 容易出错，需要审查 |
| 重要代码实现 | ✅ 适用 | 影响范围大 |
| 架构设计 | ✅ 适用 | 决策关键 |
| 安全审计 | ✅ 适用 | 风险高 |
| 简单文本生成 | ❌ 不适用 | 成本收益比低 |
| 日常对话 | ❌ 不适用 | 无需审查 |

### 2.2 模型选择原则

```
执行者选择：
├── 任务复杂度适中
├── 成本考虑
├── 速度要求
└── 基础能力满足需求

审查者选择：
├── 能力强于执行者
├── 专项能力强（如安全审查）
├── 推理能力突出
└── 可靠性高
```

## 三、审查流程

### 3.1 标准流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Agent A    │────→│  Agent B    │────→│    决策     │
│  (执行者)   │     │  (审查者)   │     │  通过/返工  │
│  基础模型   │     │  高性能模型 │     │             │
└─────────────┘     └─────────────┘     └─────────────┘
       │                   │                   │
       │                   │                   │
       ▼                   ▼                   ▼
   产出初稿          审查并反馈           判断结果
                         │
           ┌─────────────┼─────────────┐
           │             │             │
           ▼             ▼             ▼
        无问题      小问题返回      大问题重做
        通过        Agent A修改     Agent B写正确版本
```

### 3.2 详细步骤

```
Step 1: Agent A 产出初稿
├── 理解需求
├── 生成方案
├── 输出初稿
└── 附带说明

Step 2: Agent B 审查
├── 检查正确性
├── 检查完整性
├── 检查规范性
├── 检查安全性
└── 输出审查报告

Step 3: 结果处理
├── 无问题 → 直接采用
├── 小问题 → Agent A 修改 → Agent B 再审
└── 大问题 → Agent B 重写 → 直接采用
```

## 五、审查报告格式

### 5.1 标准格式

```markdown
# 审查报告

## 基本信息
- 审查对象: [文件/模块名称]
- 执行者: Agent A ([模型名称])
- 审查者: Agent B ([模型名称])
- 审查时间: [时间]

## 审查结果
- 总体评价: ✅ 通过 / ⚠️ 需修改 / ❌ 需重做
- 问题数量: 严重 [X] 个，一般 [X] 个，建议 [X] 个

## 问题列表

### 严重问题 (必须修复)
1. [问题描述]
   - 位置: [文件:行号]
   - 原因: [问题原因]
   - 建议: [修复建议]

### 一般问题 (建议修复)
1. [问题描述]
   - 位置: [文件:行号]
   - 建议: [改进建议]

### 优化建议 (可选)
1. [建议描述]

## 审查结论
[通过/需修改后通过/需重做]
```

### 5.2 快速格式（简单任务）

```markdown
# 快速审查报告

**对象**: [名称]
**结果**: ✅通过 / ⚠️需修改 / ❌需重做

**问题**:
1. [问题] - [严重程度] - [建议]

**结论**: [一句话]
```

## 六、决策规则

### 6.1 通过条件

```
┌─────────────────────────────────────────────────────────────┐
│ 通过条件                                                    │
├─────────────────────────────────────────────────────────────┤
│ ✅ 无严重问题                                               │
│ ✅ 一般问题 ≤ 3个                                          │
│ ✅ 所有功能正确实现                                         │
│ ✅ 符合项目规范                                             │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 修改条件

```
┌─────────────────────────────────────────────────────────────┐
│ 需修改后通过                                                │
├─────────────────────────────────────────────────────────────┤
│ ⚠️ 严重问题 = 0                                             │
│ ⚠️ 一般问题 > 3 或 < 10                                    │
│ ⚠️ 问题可快速修复                                          │
│ ⚠️ 修改后无需重新审查完整内容                               │
└─────────────────────────────────────────────────────────────┘
```

### 6.3 重做条件

```
┌─────────────────────────────────────────────────────────────┐
│ 需要重做                                                    │
├─────────────────────────────────────────────────────────────┤
│ ❌ 严重问题 ≥ 1                                             │
│ ❌ 一般问题 ≥ 10                                            │
│ ❌ 方向性错误                                               │
│ ❌ 大部分需要重写                                           │
│ ❌ 修复成本高于重做                                         │
└─────────────────────────────────────────────────────────────┘
```

## 七、异常处理

### 7.1 审查分歧

```
当 Agent A 和 Agent B 意见不一致时：

1. 明确分歧点
   - 具体是什么问题
   - 各自的理由是什么

2. 提交给更高层级
   - 门下省最终裁决
   - 或人工介入

3. 记录分歧
   - 分歧内容
   - 最终决策
   - 决策理由
```

### 7.2 多次返工

```
当修改次数超过限制时：

限制: 最多返工 3 次

超过限制的处理：
1. 触发告警
2. 通知调度层
3. 人工介入
4. 分析原因
   - 需求不明确？
   - 能力不匹配？
   - 沟通问题？
```

## 八、效率优化

### 8.1 审查范围优化

```
不必每次全量审查：

首次审查: 全量
后续审查: 只审查修改部分

增量审查规则：
├── 新增内容: 必须审查
├── 修改内容: 必须审查
├── 删除内容: 检查影响范围
└── 未修改: 跳过
```

### 8.2 并行审查

```
对于大型任务，可以分模块并行审查：

┌─────────────┐
│   模块 A    │────→ Agent B1 审查
├─────────────┤
│   模块 B    │────→ Agent B2 审查
├─────────────┤
│   模块 C    │────→ Agent B3 审查
└─────────────┘
         │
         ▼
    汇总审查结果
```
