# 模块化原则

> **版本**: 1.0.0
> **用途**: 多Agent项目开发模块化设计规范

---

## 一、核心原则

### 1.1 单一职责

```
┌─────────────────────────────────────────────────────────────┐
│ 原则定义                                                    │
├─────────────────────────────────────────────────────────────┤
│ 每个模块只负责一个功能，每个Agent只做一件事。              │
└─────────────────────────────────────────────────────────────┘

正确示例：
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 代码生成器  │    │ 代码审查器  │    │ 测试生成器  │
│ 只负责生成  │    │ 只负责审查  │    │ 只负责测试  │
└─────────────┘    └─────────────┘    └─────────────┘

错误示例：
┌─────────────────────────────────────────┐
│         代码全能处理器                   │
│  生成 + 审查 + 测试 + 部署 + 监控       │
└─────────────────────────────────────────┘
```

### 1.2 高内聚低耦合

```
高内聚：模块内部紧密相关
┌─────────────────────────────────────────┐
│ 用户认证模块                             │
│ ├─ 登录验证                             │
│ ├─ Token生成                            │
│ ├─ 权限检查                             │
│ └─ 会话管理                             │
│ （都是认证相关，紧密关联）               │
└─────────────────────────────────────────┘

低耦合：模块之间松散依赖
┌───────────┐     ┌───────────┐     ┌───────────┐
│ 用户模块   │────→│ 认证模块   │←────│ 权限模块   │
│ (独立)    │     │ (独立)    │     │ (独立)    │
└───────────┘     └───────────┘     └───────────┘
依赖通过接口，不直接依赖实现
```

### 1.3 接口优先

```
开发顺序：
1. 定义接口 → 2. 编写文档 → 3. 实现功能

接口定义示例（使用Pydantic）：
┌─────────────────────────────────────────────────────────────┐
│ from pydantic import BaseModel                              │
│ from typing import List, Optional                           │
│                                                             │
│ class TaskInput(BaseModel):                                 │
│     """任务输入接口"""                                      │
│     task_id: str                                             │
│     task_type: str                                          │
│     parameters: dict                                        │
│     priority: int = 3                                        │
│                                                             │
│ class TaskOutput(BaseModel):                                │
│     """任务输出接口"""                                      │
│     task_id: str                                             │
│     status: str                                              │
│     result: Optional[dict] = None                           │
│     error: Optional[str] = None                              │
└─────────────────────────────────────────────────────────────┘
```

### 1.4 配置外置

```
敏感信息 → .env 文件（不提交Git）
业务配置 → config/ 目录
硬编码 → ❌ 禁止

配置层级：
├── .env                    # 敏感信息（数据库密码、API密钥）
├── .env.example            # 环境变量示例（提交Git）
├── config/
│   ├── settings.py         # 主配置
│   ├── agents.json         # Agent配置
│   ├── models.json         # 模型配置
│   └── workflows/          # 工作流配置
│       └── default.yaml
```

## 三、接口设计

### 3.1 模块间通信

```python
# 使用消息队列解耦
class AgentMessage:
    """Agent间消息格式"""
    sender: str              # 发送者ID
    receiver: str             # 接收者ID
    message_type: str         # 消息类型
    content: dict             # 消息内容
    timestamp: datetime        # 时间戳
    priority: int = 3         # 优先级 1-5
    requires_reply: bool = False  # 是否需要回复

# 消息类型定义
MESSAGE_TYPES = {
    # 任务相关
    "task.create": "创建任务",
    "task.dispatch": "派发任务",
    "task.update": "更新状态",
    "task.complete": "任务完成",
    
    # 审核相关
    "review.submit": "提交审核",
    "review.approve": "审核通过",
    "review.reject": "审核拒绝",
    
    # 协作相关
    "collab.request": "协作请求",
    "collab.response": "协作响应",
}
```

### 3.2 依赖注入

```python
# 使用依赖注入，便于测试和替换
class CodeGenerator:
    """代码生成器"""
    
    def __init__(
        self,
        llm_service: LLMService,      # 注入LLM服务
        storage: StorageService,        # 注入存储服务
        config: dict = None            # 注入配置
    ):
        self.llm = llm_service
        self.storage = storage
        self.config = config or {}
    
    def generate(self, spec: CodeSpec) -> Code:
        """生成代码"""
        prompt = self._build_prompt(spec)
        code = self.llm.generate(prompt)
        self.storage.save(code)
        return code

# 使用示例
llm = OpenAIService(api_key="...")
storage = FileStorage(base_path="./output")
generator = CodeGenerator(llm, storage)
```

## 四、模块边界

### 4.1 清晰边界

```
┌─────────────────────────────────────────────────────────────┐
│ 模块边界检查清单                                            │
├─────────────────────────────────────────────────────────────┤
│ ✅ 单一入口：每个模块只有一个调用入口                       │
│ ✅ 明确接口：所有交互通过接口，不直接访问内部               │
│ ✅ 数据隔离：模块内部数据不暴露给外部                       │
│ ✅ 错误封装：模块内部错误不向外传播                         │
│ ✅ 版本独立：模块可以独立升级                               │
└─────────────────────────────────────────────────────────────┘
```

### 4.2 依赖方向

```
正确的依赖方向（单向，无循环）：

core ← agents ← skills ← services ← utils
 │       │        │         │        │
 └───────┴────────┴─────────┴────────┘
              所有模块都可以依赖utils

错误的依赖方向（循环依赖）：

agents ←→ skills ←→ services
  ↑          ↓          ↑
  └──────────┴──────────┘
        循环依赖 ❌
```

## 五、模块文档

### 5.1 必需文档

每个模块必须包含：

```
module_name/
├── __init__.py          # 模块初始化
├── README.md            # 模块说明
│   ├── 功能描述
│   ├── 接口列表
│   ├── 使用示例
│   └── 依赖说明
├── main.py              # 主要实现
├── tests/               # 测试文件
│   ├── test_main.py
│   └── test_integration.py
└── docs/                # 详细文档
    ├── api.md           # API文档
    └── design.md       # 设计文档
```

### 5.2 文档模板

```markdown
# 模块名称

## 功能描述
[一句话描述模块功能]

## 接口列表
| 接口名 | 输入 | 输出 | 说明 |
|--------|------|------|------|
| func_a | dict | bool | 功能说明 |

## 使用示例
```python
from module_name import ClassA

obj = ClassA(config)
result = obj.do_something()
```

## 依赖说明
- 内部依赖: [列出依赖的其他模块]
- 外部依赖: [列出第三方库]

## 更新日志
- v1.0.0 (2026-03-24): 初始版本
```

## 六、检查清单

### 6.1 模块设计检查

```
□ 是否符合单一职责原则？
□ 模块内部是否高内聚？
□ 模块之间是否低耦合？
□ 接口是否已定义？
□ 配置是否外置？
□ 依赖方向是否正确（无循环）？
□ 是否有完整文档？
□ 是否有测试覆盖？
```

### 6.2 模块评审要点

```
□ 命名是否清晰表达功能？
□ 是否有不必要的依赖？
□ 是否有硬编码？
□ 错误处理是否完善？
□ 日志是否充分？
□ 性能是否满足要求？
□ 是否便于扩展？
```
