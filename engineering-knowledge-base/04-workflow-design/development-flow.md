## 二、详细步骤

### Step 1: 需求讨论

**参与者**: 用户 + 中书省

**输入**: 用户需求（自然语言）

**处理流程**:
```
1. 用户提出需求
2. 中书省分析需求可行性
   ├── 技术可行性
   ├── 资源可行性
   └── 时间可行性
3. 创建工程文件夹
4. 生成初步计划
5. 与用户确认需求理解
```

**输出**:
- `docs/requirements.md` - 需求文档
- `PROJECT-STRUCTURE.md` - 项目结构
- `docs/plan.md` - 初步计划

**检查点**:
```
□ 需求是否明确？
□ 是否有模糊点需要澄清？
□ 用户是否确认需求理解正确？
```

---

### Step 2: 监察启动

**参与者**: 门下省

**处理流程**:
```
1. 监察Agent启动
2. 创建监察日志文件
3. 开始记录所有操作
4. 设置监控规则
5. 预警阈值配置
```

**输出**:
- `logs/inspector.log` - 监察日志
- `config/monitor-rules.json` - 监控规则

**监控内容**:
```
┌─────────────────────────────────────────────────────────────┐
│ 监控项                                                      │
├─────────────────────────────────────────────────────────────┤
│ - 操作日志（谁在何时做了什么）                              │
│ - 决策记录（为什么做这个决定）                              │
│ - 异常情况（超时、错误、偏离）                              │
│ - 资源使用（时间、Token、存储）                             │
└─────────────────────────────────────────────────────────────┘
```

---

### Step 3: 知识准备

**参与者**: 学习部 + 记忆部

**处理流程**:
```
1. 收集相关技术文档
   ├── 官方文档
   ├── 最佳实践
   └── 类似项目
2. 创建工程知识库
   └── docs/engineering-knowledge-base/
3. 设计Agent配置
   └── config/agents.json
4. 开发必要Skills
   └── skills/
5. 建立知识关联
```

**输出**:
- `docs/engineering-knowledge-base/` - 知识库
- `config/agents.json` - Agent配置
- `skills/` - Skills模块

---

### Step 4: 计划生成

**参与者**: 中书省

**输入**:
- 需求文档
- 工程知识库
- 开发规范

**处理流程**:
```
1. 读取知识库内容
2. 分析开发规范
3. 任务分解
   ├── 功能模块
   ├── 接口设计
   ├── 数据设计
   └── 部署设计
4. 分配给各部门
5. 生成详细时间线
```

**输出**:
- `docs/detailed-plan.md` - 详细计划
- `docs/task-assignments.md` - 任务分配
- `docs/timeline.md` - 时间线

---

### Step 5: 架构审核

**参与者**: 门下省

**审核内容**:
```
1. 技术方案可行性
   ├── 技术栈选择是否合理
   ├── 架构设计是否合理
   └── 是否有技术风险

2. 模块划分合理性
   ├── 是否符合单一职责
   ├── 是否高内聚低耦合
   └── 接口设计是否清晰

3. 性能评估
   ├── 是否有性能瓶颈
   ├── 扩展性是否考虑
   └── 资源使用是否合理

4. 完整性检查
   ├── 是否覆盖所有需求
   ├── 异常处理是否完善
   └── 边界条件是否考虑
```

**输出**:
- `docs/architecture.md` - 架构文档
- `docs/tech-stack.md` - 技术选型
- `docs/api-design.md` - 接口设计

**决策**:
```
审核结果:
├── 通过 → 进入Step 6
├── 需修改 → 返回Step 4修改
└── 不通过 → 终止或重新规划
```

---

### Step 6: 安全审计

**参与者**: 刑部

**审计内容**:
```
1. 代码安全扫描
   ├── 注入漏洞
   ├── 认证问题
   └── 授权问题

2. 敏感信息检查
   ├── 硬编码密钥
   ├── 日志泄露
   └── 配置暴露

3. 权限设计审核
   ├── 最小权限原则
   ├── 权限分离
   └── 默认拒绝

4. 输入验证检查
   ├── 参数校验
   ├── 类型检查
   └── 边界检查
```

**输出**:
- `reports/security-audit.md` - 安全报告
- `reports/fixes.md` - 修复记录

---

### Step 7: 代码执行

**参与者**: 兵部（主导）+ 其他部门配合

**执行流程**:
```
1. 按计划执行开发任务
   ├── 吏部: Agent配置
   ├── 户部: 数据处理
   ├── 礼部: 文档编写
   ├── 兵部: 代码开发
   ├── 刑部: 代码审查
   └── 工部: 部署配置

2. 遵循编码规范
3. 编写单元测试
4. 代码评审
5. 提交代码
```

**输出**:
- `modules/` - 源代码
- `tests/` - 测试代码
- `docs/` - 更新文档

---

### Step 8: 调试验证

**参与者**: 中书省

**处理流程**:
```
1. 集成测试
   ├── 功能测试
   ├── 接口测试
   └── 性能测试

2. 问题定位
   ├── 分析日志
   ├── 复现问题
   └── 定位原因

3. 协调修复
   ├── 分配给对应部门
   └── 跟踪修复进度

4. 回归测试
   └── 验证修复效果
```

**输出**:
- `reports/test-report.md` - 测试报告
- `reports/bugs.md` - 问题清单

---

### Step 9: 需求验收

**参与者**: 门下省

**验收流程**:
```
1. 对照原始需求
   ├── 逐项检查
   └── 功能完整性

2. 质量检查
   ├── 代码质量
   ├── 文档质量
   └── 测试覆盖率

3. 性能指标
   ├── 响应时间
   ├── 吞吐量
   └── 资源占用

4. 生成验收报告
```

**输出**:
- `reports/acceptance.md` - 验收报告
- `reports/pending.md` - 遗留问题

**验收标准**:
```
┌─────────────────────────────────────────────────────────────┐
│ 通过条件                                                    │
├─────────────────────────────────────────────────────────────┤
│ ✅ 所有需求功能已实现                                       │
│ ✅ 无严重Bug                                                │
│ ✅ 测试覆盖率 > 80%                                         │
│ ✅ 性能指标达标                                             │
│ ✅ 文档完整                                                 │
└─────────────────────────────────────────────────────────────┘
```

---

### Step 10: 用户交付

**参与者**: 用户 + 中书省

**交付流程**:
```
1. 功能演示
   ├── 演示环境准备
   ├── 功能演示
   └── 使用说明

2. 用户测试
   ├── 用户试用
   └── 收集反馈

3. 问题记录
   ├── 功能问题
   ├── 易用性问题
   └── 改进建议

4. 确认交付
```

**输出**:
- `docs/demo.md` - 演示文档
- `feedback/user-feedback.md` - 用户反馈

---

**参与者**: 工部

**运维流程**:
```
1. 部署上线
   ├── 环境配置
   ├── 服务部署
   └── 数据迁移

2. 监控配置
   ├── 性能监控
   ├── 错误监控

3. 日志收集
   ├── 应用日志

   ├── 性能优化

```

**输出**:
- `docs/deployment.md` - 部署文档

## 三、状态转换

### 3.1 任务状态

```
draft → planning → review → dispatch → execute → check → accept → done
                  ↑         │
                  └── reject ← (封驳返回)
```

### 3.2 状态与步骤对应

| 状态 | 对应步骤 | 负责部门 |
|------|----------|----------|
| draft | Step 1 | 中书省 |
| planning | Step 4 | 中书省 |
| review | Step 5, 6 | 门下省、刑部 |
| reject | Step 5 (不通过) | 门下省 |
| dispatch | Step 7开始 | 尚书省 |
| execute | Step 7 | 六部 |
| check | Step 8, 9 | 中书省、门下省 |
| accept | Step 9通过 | 门下省 |
| done | Step 10, 11 | 工部 |

## 四、异常处理

### 4.1 常见异常

| 异常 | 发生阶段 | 处理方式 |
|------|----------|----------|
| 需求变更 | 任何阶段 | 回到Step 1重新评估 |
| 技术障碍 | Step 7 | 中书省协调，必要时调整方案 |
| 质量不达标 | Step 9 | 返回Step 7修复 |
| 超时 | 任何阶段 | 门下省告警，人工介入 |
| 资源不足 | Step 3, 7 | 中书省协调资源 |

### 4.2 回滚机制

```
┌─────────────────────────────────────────────────────────────┐
│ 回滚触发条件                                                │
├─────────────────────────────────────────────────────────────┤
│ - 方案审核不通过超过3次                                     │
│ - 安全审计发现严重漏洞                                      │
│ - 测试通过率低于50%                                         │
│ - 用户明确拒绝                                              │
└─────────────────────────────────────────────────────────────┘

回滚流程:
1. 记录当前状态
2. 恢复到上一个稳定版本
3. 分析失败原因
4. 调整方案
5. 重新执行
```
